## 중요 이벤트 (당일): GitHub Pages 직후 404

### 현상

- `git push origin main`은 **성공**했고, `main` 브랜치에는 `unit36/index.html` 등이 **즉시 반영**됨(예: `raw.githubusercontent.com/.../unit36/index.html`로 확인 가능).
- 그러나 **푸시 직후·수십 초~1분 가량** `https://word1600.github.io/game/unit36/` 요청 시 **HTTP 404**가 관측됨.
- 일정 시간 후 동일 URL은 **200 및 정상 HTML**로 응답했고, 사용자 측에서도 **잘 실행됨**을 확인함.

### 원인 정리 (재발 방지용)

- GitHub Pages는 **저장소 `main`과 즉시 동일하게 보이지 않을 수 있음**(빌드·CDN 캐시·엣지 전파 지연).
- 따라서 **“푸시 성공 = 곧바로 Pages에서 열림”을 단정할 수 없고**, 반대로 **“404 = 푸시 실패”로 단정하면 안 됨**.

### 권장 확인 순서

1. **`git push origin main` 성공 여부**로 “원격 반영”을 판단한다.
2. Pages URL이 404면 **수 분 간격으로 재시도**하고, 필요 시 **`raw.githubusercontent.com`으로 파일 존재**를 확인한다.
3. 사용자에게는 **배포 지연 가능**을 미리 안내한다(`Unit21_자동화_배포_가이드.md`와 동일 취지).

### 금지 (재발 방지)

- **`git status`만으로** “아직 배포 안 됨”을 단정하지 않는다(규칙 문서와 동일).
- **직후 404만 보고** 푸시·배포 파이프라인 전체를 실패로 오인하지 않는다.

## 부가 이슈 2: `enhanced_validator.py` (Windows 콘솔)

### 현상

- `python enhanced_validator.py data/unit36.json` 실행 시, 이모지 등 유니코드 출력에서 **`UnicodeEncodeError: 'cp949' codec can't encode...`** 발생 가능.

### 참고

- 데이터 파일 자체 오류라기보다 **Windows 기본 코드페이지(cp949)와 콘솔 출력** 조합 이슈에 가깝다.
- 검증이 필요하면 `PYTHONIOENCODING=utf-8` 설정 또는 UTF-8 콘솔 사용 등을 검토할 수 있다(당일 작업 완료에는 필수 아님).
